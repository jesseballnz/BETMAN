# AI, Model Tuning & Bakeoff Audit — 27 Mar 2026

## Scope
Re-audit of the Model Bakeoff evaluation framework, individual runner analysis depth, and production-readiness checks. Builds on the 26 Mar 2026 audit.

## Findings & Fixes Applied

### 1. Bakeoff — No Gate for JSON Parse Failures (HIGH · FIXED)

**Finding:** When `/api/ask-selection` returned a non-JSON body (e.g. a plain-text error page, an HTML 502 from a proxy, or a truncated response), `model_bakeoff.js` swallowed the parse failure silently via `.catch(() => ({}))`. The resulting row had `ok=false` but `error=null` and `jsonParseError` was absent — making it impossible to distinguish a JSON parse failure from an HTTP error or a legitimate `ok:false` API response.

Additionally, when the API returned HTTP 200 with `ok:false` (an application-level error), no diagnostic was emitted, so the failure cause was opaque in the JSONL output.

**Fixes applied (`scripts/model_bakeoff.js`):**
- Added `jsonParseError` boolean field to every row in the JSONL output. When the response body cannot be parsed as JSON, `jsonParseError: true` is set and a diagnostic line is written to `stderr`: `json_parse_error model=<model> prompt=<id> status=<http_status> :: <error_message>`.
- When HTTP 200 but `ok:false`, a separate `stderr` line is written: `api_ok_false model=<model> prompt=<id> :: <error_or_fallback_reason>`.
- `jsonParseError` is tracked per-model in the aggregation step as `parseErrors` → surfaced as `parseErrorRate %` in both the leaderboard JSON and the markdown table.
- New column `Parse Err %` added to the leaderboard markdown table.

### 2. Bakeoff — `scoreAnswer` Has No Format-Compliance Gate (MEDIUM · FIXED)

**Finding:** `scoreAnswer()` used keyword matching only. The `decision-format` prompt requires a strict pipe-delimited output (`Decision | Stake | Confidence% | Invalidation`). A model could score 100 by including all four keywords in free-form prose, even though the output was not machine-readable. Similarly, the `race-win-core` and `runner-profile-depth` prompts benefit from checking whether the model actually named the specific runners from the context.

**Fixes applied (`scripts/model_bakeoff.js`):**
- Added `requirePipeFormat: true` prompt flag. When set, `scoreAnswer` applies a 25-point penalty if the answer does not contain a pipe-delimited line matching `word | word | word | word`.
- Added `requireRunnerMentions: [name, ...]` prompt flag. When set, `scoreAnswer` checks what fraction of the listed runner names appear in the answer. If fewer than 50% are mentioned, a 20-point penalty is applied.
- `scoreAnswer` and `isTrueFallback` are now exported via `module.exports` when the file is `require()`-d (not run directly), enabling unit testing without spawning a live server.

### 3. Bakeoff — No Individual Runner Analysis Prompt (MEDIUM · FIXED)

**Finding:** The 8 existing bakeoff prompts tested race-level outputs (win chances, multis, stake sizing, decision format) but none specifically exercised the model's ability to produce a per-runner profile covering every data field that BETMAN sends in `RACE_FIELD_DATA` (formIndicators, lastStarts, gear, stats, apprentice, loveracingNote, etc.). This left the quality of individual runner analysis unverified.

**Fixes applied:**
- Added new prompt `runner-profile-depth` to `bakeoff/prompts.json` with `requireRunnerMentions` gate covering all 5 runners in the context.
- Created `bakeoff/tests/rw9-runner-profile.txt`: a full Sha Tin R11 field with rich `RACE_FIELD_DATA` JSON (all fields populated: `formIndicators`, `lastStarts`, `gear`, `stats`, `apprentice`, `loveracingNote`, `speedmap`, pedigree, odds, trainer location).
- The new prompt validates that the model: (a) names each runner individually, (b) references barrier/jockey/trainer/suitability/speedmap/sire keywords, and (c) names a top-2 with an edge rationale.

### 4. System Prompt — `RACE_FIELD_DATA` Field Descriptions Incomplete (LOW · FIXED)

**Finding:** Rule 11 of `BETMAN_ANALYST_SYSTEM_PROMPT` instructed the AI to "populate full-field horse profiles" from `RACE_FIELD_DATA` but only explained `stats` and `form`. The `formIndicators`, `lastStarts`, `apprentice`, and `loveracingNote` fields were not described, so models with weaker instruction-following would skip or misinterpret them.

**Fix:** Rule 11 now explicitly describes all key fields:
- `formIndicators` — contains flags (e.g. `Early speed: HIGH`, `Wet track: STRONG`) to use for map fit and track condition assessment.
- `lastStarts` — array of recent finish/distance/track/condition entries for consistency and trend analysis.
- `loveracingNote` — official form notes; treat as additional source commentary.
- `apprentice` — when populated (e.g. `"-2kg"`), factor the weight reduction into the runner assessment.

## Recommendations (Deferred)

1. **Composite score penalise parse errors** — Consider subtracting from composite score when `parseErrorRate > 0`. Currently parse errors reduce `successRate` (which reduces composite indirectly) but are not called out explicitly in the score.
2. **Semantic answer scoring** — `scoreAnswer()` uses keyword + format matching only. LLM-as-judge scoring would better capture analytical quality.
3. **Meeting profile in general queries** — Currently only included for specific race context.
4. **User notes persistence** — Currently localStorage only. Consider syncing to server for cross-device access.
